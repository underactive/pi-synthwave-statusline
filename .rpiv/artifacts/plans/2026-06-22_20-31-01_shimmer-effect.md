# Shimmer Effect Implementation Plan

## Overview
Add a per-character ANSI color cycling shimmer effect to the model name and provider text in the Synthwave statusline footer. The shimmer activates on `/new` or `/reload` (via `session_start`), runs for 10 seconds at ~12.5fps using `setInterval`, then settles back to static Synthwave magenta colors. Follows the existing closure-based timer pattern from the TPS implementation and the per-character color wrapping pattern from the Pi SDK's rainbow-editor example.

## Requirements
- Shimmer applies to model name + provider text only (not thinking level)
- Activates after `/new` or `/reload` (session_start events)
- Runs for exactly 10 seconds, then settles to static colors
- Per-character color wave effect cycling through 8 Synthwave 84 palette colors
- Timer must be properly cleaned up in session_shutdown and dispose()
- No performance impact on streaming or rendering

## Desired End State

After `/new` or `/reload`, the model name and provider in the statusline footer shimmer with a flowing per-character color wave for 10 seconds:

```
Before shimmer (static):
  (anthropic) claude-sonnet-4-20250514  🧠 high

During shimmer (animated):
  (anthropic) claude-sonnet-4-20250514  🧠 high
  ↑ each character cycles through magenta → cyan → pink → green → yellow → orange → lime → light cyan
  ↑ wave offset creates flowing left-to-right motion

After 10s (settles back to static):
  (anthropic) claude-sonnet-4-20250514  🧠 high
```

## What We're NOT Doing
- **Handler accumulation fix** — Pre-existing bug where `pi.on()` calls accumulate across session transitions. Tracked separately.
- **Model-remind notification merge** — The `ctx.ui.notify()` pattern from model-remind is not being added to statusline. Statusline already hooks `session_start`; the shimmer is the only addition.
- **Thinking level shimmer** — The ORANGE thinking level block stays static.
- **Configurable duration/colors** — Hardcoded 10s duration and 8-color palette. No user configuration.
- **Pause/resume on `/undo`** — `session_start` does not fire on `/undo`; shimmer only triggers on session context changes.

## Decisions

### Decision 1: Timer lifecycle — follow TPS pattern
**Ambiguity**: Where to start the shimmer timer and how to manage its lifecycle.
**Explored**:
- Option A: Start timer in `session_start` entry, use `requestFooterRender` for render trigger — but `tui` is not available yet at this point
- Option B: Start timer in `setFooter` callback where `tui.requestRender()` is captured — matches TPS pattern at `statusline.ts:170-176`
**Decision**: Option B — `setFooter` callback. This is where `requestFooterRender` is set (line 138), and the timer needs `tui.requestRender()` to drive renders. Cleanup in both `session_shutdown` (line 125) and `dispose()` return (line 176).
**Evidence**: `statusline.ts:170-176` — existing pattern for timer lifecycle in setFooter

### Decision 2: Color palette — 8 Synthwave 84 colors from C object
**Decision**: Use 8 colors from the existing `C` object: MAGENTA, CYAN, CAC (hot pink), GREEN, YELLOW, ORANGE, API (lime green), CYAN_LIGHT. No new color constants needed.
**Evidence**: `statusline.ts:1-66` — all shimmer colors already defined

### Decision 3: Per-character wave formula
**Decision**: Phase offset of 75ms per character position, 600ms full color cycle. Formula: `(elapsedMs + charOffset * 75) % 600` maps to color index.
**Evidence**: Research artifact — tested formula produces visible wave at ~12.5fps

### Decision 4: Shimmer scope — model name + provider only
**Decision**: Shimmer applies to `modelDisplayFinal` text (provider + model name). The thinking level block (`modelBlock`) stays static. The `visibleWidth()` function handles ANSI codes correctly, so shimmer doesn't affect layout.
**Evidence**: `statusline.ts:245-249` — modelDisplayFinal construction; `statusline.ts:242-258` — line1 layout uses visibleWidth

### Decision 5: Handler accumulation — out of scope
**Decision**: Pre-existing bug where `pi.on()` calls accumulate across session transitions. Not introduced by shimmer, not fixed in this change. Tracked separately.
**Evidence**: Research artifact Open Question; loader behavior in pi-coding-agent

```typescript
/** Per-character color-cycling shimmer effect for model name display */
function shimmerString(text: string, elapsedMs: number): string {
	return [...text]
		.map((ch, i) => {
			const phase = (elapsedMs + i * 75) % SHIMMER_CYCLE_MS;
			const colorIndex = Math.floor((phase / SHIMMER_CYCLE_MS) * SHIMMER_COLORS.length);
			return `${SHIMMER_COLORS[colorIndex]}${ch}`;
		})
		.join("") + C.RESET;
}
```

```typescript
			requestFooterRender = () => tui.requestRender();
			const unsub = footerData.onBranchChange(() => tui.requestRender());

			// Start shimmer effect for model name
			shimmerActiveSince = Date.now();
			if (shimmerTimer) clearInterval(shimmerTimer);
			shimmerTimer = setInterval(() => {
				const elapsed = Date.now() - shimmerActiveSince;
				if (elapsed > SHIMMER_DURATION) {
					clearInterval(shimmerTimer!);
					shimmerTimer = null;
				}
				requestFooterRender?.();
			}, SHIMMER_INTERVAL);

			return {
				dispose: () => {
					unsub();
					if (shimmerTimer) {
						clearInterval(shimmerTimer);
						shimmerTimer = null;
					}
				},
```

## Performance Considerations
- 80ms setInterval (~12.5fps) — well below TUI's 16ms minimum render interval
- `requestRender()` coalesces multiple calls per tick — shimmer timer and streaming events don't cause redundant renders
- Per-character ANSI wrapping adds ~N*15 bytes per render where N = model name + provider length (~30-50 chars) — negligible
- Auto-terminating after 10s ensures no long-running timer overhead
