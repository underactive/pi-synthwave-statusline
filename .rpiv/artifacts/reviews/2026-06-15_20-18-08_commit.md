## ðŸ”´ Critical

### I1 ðŸ”´ Cross-branch context data displayed as current-branch data `[cascade: stranded-state]`

**Where**
`statusline.ts:142` â€” `for (const entry of ctx.sessionManager.getEntries()) {`
`statusline.ts:147` â€” `totalCacheWrite += m.usage.cacheWrite;`
`statusline.ts:169` â€” `const promptTotal = totalInput + totalCacheRead + totalCacheWrite;`

**Code**
```ts
// Line 142: reads ALL branches
for (const entry of ctx.sessionManager.getEntries()) {
// ...
    totalCacheWrite += m.usage.cacheWrite;
}

// Line 169: denominator includes cache writes, inflating promptTotal
const promptTotal = totalInput + totalCacheRead + totalCacheWrite;
```

**Why**
Three additive defects combine to produce silently wrong footer stats when the user has branched the conversation. (1) `getEntries()` returns messages across all session branches; peer extensions (`tool-counter.ts`, `coms.ts`) correctly use `getBranch()` for current-branch iteration. (2) `totalCacheWrite` is added to `promptTotal` in the cache-hit ratio denominator (`totalCacheRead / promptTotal`), inflating the divisor beyond the semantic cache-hit computation. (3) When `promptTotal === 0` but `totalCost > 0` (zero-input-cost model calls), `cacheHitStr` falls through the `promptTotal > 0` guard to `"-"`, but cost still displays â€” creating an undetectable inconsistency. The user sees plausible-seeming stats that are mathematically meaningless (hybrid of current and stale branch data, wrong denominator).

**Fix**
Use `getBranch()` instead of `getEntries()` to scope accumulation to the active branch. Remove `totalCacheWrite` from the `promptTotal` denominator â€” compute cache-hit ratio as `totalCacheRead / (totalCacheRead + totalInput)` if the intent is cache-read efficiency, or omit the ratio entirely if unclear.

**Alt**
If cross-branch totals are the intended display, label them explicitly (e.g. "all-branch cache:") so the user can distinguish from current-branch context usage.

### I4 ðŸŸ¡ `"?%"` context-unknown path unreachable when `getContextUsage()` returns `undefined`

**Where**
`statusline.ts:155-157`

**Code**
```ts
const contextWindow = contextUsage?.contextWindow ?? ctx.model?.contextWindow ?? 0;
const contextPctValue = contextUsage?.percent ?? 0;
const contextPctStr = contextUsage?.percent !== null
    ? `${Math.round(contextPctValue)}%`
    : "?%";
```

**Why**
When `contextUsage` is `undefined` (early session init or missing data), `contextUsage?.percent` evaluates to `undefined`, and `undefined !== null` is `true` â€” so the truthy branch runs, displaying `"0%"` instead of `"?"`. The "?%" sentinel is only reachable when `contextUsage` is an object with `percent` explicitly set to `null`. The `?? 0` fallback at line 156 silently conflates `undefined` context with `null` percent, losing the "unknown" signal.

**Fix**
Check `contextUsage == null` (catches both `undefined` and `null`) before accessing `.percent`. Show "?%" when contextUsage is absent:
```ts
const contextPctStr = contextUsage == null || contextUsage.percent == null
    ? "?%"
    : `${Math.round(contextUsage.percent)}%`;
```

### I7 ðŸŸ¡ Cache-write tokens included in cache-hit ratio denominator

**Where**
`statusline.ts:169-170` â€” `promptTotal = totalInput + totalCacheRead + totalCacheWrite`

**Code**
```ts
const promptTotal = totalInput + totalCacheRead + totalCacheWrite;
// ...
((totalCacheRead / promptTotal) * 100).toFixed(1) + "%"
```

**Why**
The cache-hit ratio divides `totalCacheRead` by `promptTotal` which includes `totalCacheWrite`. This deflates the displayed percentage below the actual cache-read rate. Peer extensions (`exit-stats.ts:337-338`) read both `cacheRead` and `cacheWrite` as separate displayed values but do not combine them in a ratio. The expected semantic for a "cache hit" indicator is `cacheRead / totalPromptTokens` (where prompt tokens = input + cacheRead), not `cacheRead / (input + cacheRead + cacheWrite)`.

**Fix**
Compute `promptTotal` as `totalInput + totalCacheRead` for the cache-hit ratio, or display cache writes separately as a raw count alongside cache reads.

### I10 ðŸŸ¡ Display inconsistency when `totalCost > 0` but `promptTotal === 0`

**Where**
`statusline.ts:169-172` â€” `promptTotal > 0` guard, `statusline.ts:253` â€” cost always pushed

**Code**
```ts

    ? `...${((totalCacheRead / promptTotal) * 100).toFixed(1) + "%"}`
```

**Why**
When `promptTotal === 0` (zero input/cache tokens) but `totalCost > 0` (model charges a flat per-call fee), the cache-hit display shows `"-"` while cost shows a non-zero value and `totalCacheRead` shows `"0"`. **Fix**
Hide the cache-hit segment entirely or display `"n/a"` when `promptTotal === 0`, and ensure `formatTokens(0)` renders as `"0"` consistently.
